# agent-unity-wrapper

Pure skill plugin — no binary of its own. It wraps the **external Unity MCP server**: a skill teaches Claude how to drive it, and (once wired) the manifests point at that MCP. This repo ships **only** the skill + manifests; the Unity MCP itself is a separate, pre-existing server.

> **Wired.** The Unity MCP (`mcpforunityserver==9.7.1`) is now wired into both manifests under the server key `unityMCP`. The scaffold state is resolved: `mcpServers` is present inline in both `.claude-plugin/plugin.json` and `.codex-plugin/plugin.json`, and `skills/unity-wrapper/SKILL.md` contains the full tool inventory and usage docs.

## Contracts an agent won't infer from the tree

- **The wrapped MCP is external and inline, not a `.mcp.json`.** When wired, the MCP command goes into an inline `mcpServers` block in **both** `.claude-plugin/plugin.json` and `.codex-plugin/plugin.json` (the two hosts can't share one external file — `${CLAUDE_PLUGIN_ROOT}` doesn't expand under Codex). Model it on `agent-serena-wrapper`, whose Claude manifest uses `--context claude-code` + `${CLAUDE_PROJECT_DIR}` and whose Codex manifest uses `--context codex` + `--project-from-cwd`.
- **Release is orphan-branch + marketplace dispatch.** `release.yml` (manual: Actions → release → `version=X.Y.Z`) stamps the version into both manifests, force-pushes an orphan `release` branch holding only install-ready files, and POSTs a dispatch (`category: skill`) to `Seretos/agent-marketplace`. `main` and `release` share no history. Clients install at the tag `agent-unity-wrapper--vX.Y.Z`.
- **Required secret:** `MARKETPLACE_DISPATCH_TOKEN` — fine-grained PAT, `Contents: RW` + `Pull requests: RW` on `Seretos/agent-marketplace` only.
- **Per-worktree Unity instances = status-dir isolation + a coupling to `agent-worktree`.** Each worktree drives its own Unity Editor with no "select instance" UI step, because both halves of the Unity MCP point at the same worktree-local `UNITY_MCP_STATUS_DIR`:
  - The manifests set it in `unityMCP.env` — `${CLAUDE_PROJECT_DIR}/.unity-mcp` (Claude) and cwd-relative `.unity-mcp` (Codex; Codex has no `${CLAUDE_PROJECT_DIR}`, and the Python server resolves a relative value against its working dir = the worktree root). Both the C# bridge and Python server use the value **literally** (no `~`/relative rewriting on the Unity side), so the editor launch must pass an **absolute** path.
  - The launch half lives in the **target Unity repo**, not here: `scripts/prepare-unity-worktree.ps1` writes a managed `start:`/`stop:` block into that repo's `.seretos/worktree-setup.yml` (the `agent-worktree` contract — this is the explicit cross-plugin coupling), adds `com.coplaydev.unity-mcp` to `Packages/manifest.json`, and ignores `.unity-mcp/`. `worktree_start`/`worktree_stop` then bring a headless Unity up/down per worktree. The script is idempotent and never clobbers a foreign `start:`/`stop:` block. `worktree_start` was chosen over `vdesktop` `launch_app` because that tool can't inject per-worktree env (agent-vdesktop#59 / lib-python-vdesktop#28).
  - **Bridge auto-start is NOT committable.** The bridge's `AutoStartOnLoad` is a Unity *EditorPref* (per-machine, hashed value name) — it does not survive a fresh clone and can't be set reliably outside Unity. So the start step boots the bridge deterministically via `-executeMethod MCPForUnity.Editor.McpCiBoot.StartStdioForCi` + `UNITY_MCP_ALLOW_BATCH=1` (modeled on CoplayDev's own CI) instead of relying on EditorPrefs.
  - **Flagged open items (per ticket #3, not pre-solved):** Unity editor binary resolution defaults to `UNITY_EDITOR_PATH`, else Unity Hub default install path derived from `ProjectVersion.txt` (OS-dependent); the bridge-package git pin (`#v9.7.1`) should track the server pin (`mcpforunityserver==9.7.1`); first-open `Library/` re-import is the real per-worktree resource cost (consider capping concurrent Unity-enabled worktrees); the launch is **headless** (`-batchmode -nographics`) by default — switch to a GUI editor by editing the managed block if interactive editing is needed.
- **`assets/icon.png` and `description.md` are release artifacts, not just repo files.** The dispatch payload sends `raw.githubusercontent.com/${repo}/${TAG}/assets/icon.png` and `.../description.md` URLs to the marketplace, so both must live on the orphan `release` branch at the tagged commit — `release.yml`'s stage step copies them into the staging tree. Ship a real icon and a filled-in `description.md` before cutting v0.0.1 or the listing has no image / placeholder text.
